# Provisioning Foundation — Implementation Plan

**Milestone:** 5 – Phase A (Architectural Foundation)
**For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Define Provisioning as a first-class platform capability with a complete domain specification. This is an architectural foundation exercise — no UI implementation. Phase B builds the operator console on top of this established domain model.

**Domain-first principle:** The domain model drives the backend contract — not the reverse. All RPCs, tables, and service wrappers are derived from the domain specification, not designed alongside it.

**Authorization model:** All provisioning RPCs enforce platform-operator authorization internally (via `is_platform_operator()`). Service wrappers are thin pass-throughs that call RPCs without adding business logic.

## Specification (pre-implementation)

### 1. Domain Model

#### 1.1 ProvisioningRequest

A user-initiated request to provision a new target. Represents the intent, not the execution.

```
ProvisioningRequest
 ├── correlationId: string (stable, idempotent, generated by caller)
 ├── targetType: string (extensible — 'workspace' initially)
 ├── targetConfig: jsonb (target-specific parameters)
 ├── requestedBy: uuid (auth.users)
 ├── requestedAt: timestamptz
 └── status: ProvisioningRequestStatus
```

#### 1.2 ProvisioningJob

An execution unit spawned from a request. One request produces exactly one job (no fan-out). The job tracks the lifecycle of a single provisioning attempt.

```
ProvisioningJob
 ├── id: uuid (PK)
 ├── requestId: uuid → ProvisioningRequest.id (FK)
 ├── targetType: string
 ├── targetId: uuid (the entity being provisioned)
 ├── status: ProvisioningJobStatus
 ├── currentStage: string (which stage of the pipeline is active)
 ├── attemptCount: integer (starts at 1, increments on retry)
 ├── ownerId: uuid (current operator handling this job — nullable)
 ├── requestedBy: uuid
 ├── completedBy: uuid (nullable)
 ├── requestedAt: timestamptz
 ├── startedAt: timestamptz (nullable)
 ├── completedAt: timestamptz (nullable)
 ├── createdAt: timestamptz (row creation)
 └── updatedAt: timestamptz (last status change)
```

#### 1.3 ProvisioningStage

A named phase within a provisioning pipeline. Stages are sequential and pipeline-specific.

```
ProvisioningStage
 ├── jobId: uuid → ProvisioningJob.id (FK)
 ├── stageName: string (e.g. 'validate_config', 'create_schema', 'seed_data')
 ├── status: StageStatus
 ├── startedAt: timestamptz (nullable)
 ├── completedAt: timestamptz (nullable)
 ├── output: jsonb (stage-specific result data)
 └── error: text (nullable)
```

#### 1.4 ProvisioningFailure

An immutable record of every failure that occurs during a provisioning attempt. Preservation of failure history is an invariant.

```
ProvisioningFailure
 ├── id: uuid (PK)
 ├── jobId: uuid → ProvisioningJob.id (FK)
 ├── stageName: string (which stage failed)
 ├── failureCode: string (machine-readable — e.g. 'SCHEMA_EXISTS', 'INVALID_CONFIG')
 ├── failureReason: text (human-readable detail)
 ├── attemptNumber: integer
 ├── failedAt: timestamptz
 └── metadata: jsonb (extensible context)
```

#### 1.5 ProvisioningTarget

A resolved reference to a concrete entity being provisioned. Decouples the provisioning system from knowing target internals.

```
ProvisioningTarget
 ├── targetType: string
 ├── targetId: uuid
 ├── displayName: string (human-readable)
 ├── targetSchema: text (nullable — only for schema-backed targets)
 └── metadata: jsonb (target-specific)
```

#### 1.6 ProvisioningAction

An operator-initiated action on a provisioning job. Every action is an auditable event.

```
type ProvisioningAction =
  | { type: 'cancel' }
  | { type: 'retry'; reason: string }
  | { type: 'assign'; ownerId: uuid }
  | { type: 'override'; field: string; value: unknown }
```

### 2. State Machine

#### 2.1 ProvisioningRequest States

```
                  ┌──────────────┐
                  │   REQUESTED  │
                  └──────┬───────┘
                         │
                    [enqueue]
                         │
                         ▼
                  ┌──────────────┐
           ┌─────▶│    QUEUED    │◀────────────┐
           │      └──────┬───────┘             │
           │             │                     │
           │        [dispatch]            [retry]
           │             │                     │
           │             ▼                     │
           │      ┌──────────────┐             │
           │      │ PROVISIONING │             │
           │      └──────┬───────┘             │
           │             │                     │
           │        ┌────┴────┐                │
           │        │         │                │
           │        ▼         ▼                │
           │  ┌─────────┐ ┌──────┐             │
           │  │SUCCEEDED│ │FAILED│─────────────┘
           │  └─────────┘ └──────┘
           │
           │    [cancel from REQUESTED]
           │         │
           │         ▼
           │  ┌────────────┐
           └──│ CANCELLED  │
              └────────────┘

              [cancel from QUEUED]
                   │
                   ▼
              ┌────────────┐
              │ CANCELLED  │
              └────────────┘
```

#### 2.2 Stage States

```
PENDING → RUNNING → SUCCEEDED
PENDING → RUNNING → FAILED
```

Stages are sequential — stage N must reach SUCCEEDED before stage N+1 enters RUNNING.

#### 2.3 Legal Transitions

| From | To | Trigger | Description |
|------|----|---------|-------------|
| REQUESTED | QUEUED | System (enqueue) | Request validated and added to work queue |
| REQUESTED | CANCELLED | Operator | Request withdrawn before execution |
| QUEUED | PROVISIONING | System (dispatch) | Worker picks up the job |
| QUEUED | CANCELLED | Operator | Cancelled before execution starts |
| PROVISIONING | SUCCEEDED | System | All stages completed successfully |
| PROVISIONING | FAILED | System | A stage failed, pipeline halts |
| FAILED | QUEUED | Operator (retry) | Operator triggers retry with incremented attempt |

#### 2.4 Forbidden Transitions

| Transition | Why Forbidden |
|------------|---------------|
| SUCCEEDED → anything | Completed jobs are immutable |
| CANCELLED → QUEUED (retry) | Cancelled jobs cannot be retried — create a new request |
| PROVISIONING → CANCELLED | System owns the PROVISIONING state — cannot cancel mid-execution |
| QUEUED → REQUESTED | No backward transitions |

---

### 3. Transition Rules

1. **REJECT_CANCELLED_RETRY:** A CANCELLED job must never transition to QUEUED. Any retry attempt on a cancelled job must be rejected.
2. **REJECT_SUCCEEDED_MUTATION:** Once SUCCEEDED, no state transition is legal. The record is immutable.
3. **REJECT_CONCURRENT_DISPATCH:** Only one job may be in QUEUED or PROVISIONING state for the same target at any time. A second dispatch for the same target must be blocked.
4. **REJECT_MISSING_TRANSITION:** Any attempt to apply an undefined transition must fail with a descriptive error.
5. **REJECT_UNOWNED_RETRY:** A retry from FAILED → QUEUED must only be initiated by an authorized platform operator.

---

### 4. Operational Invariants

| # | Invariant | Enforcement |
|---|-----------|-------------|
| I1 | Provisioning jobs are append-only. | No UPDATE on completed rows — only INSERT on new attempts. |
| I2 | Completed jobs are immutable. | SUCCEEDED and CANCELLED rows reject all UPDATEs via CHECK constraint or trigger. |
| I3 | Cancelled jobs cannot be retried. | RPC-level guard on retry: reject if previous attempt is CANCELLED. |
| I4 | Only one active provisioning job per target. | Unique partial index on (target_type, target_id) WHERE status IN ('QUEUED', 'PROVISIONING'). |
| I5 | Every provisioning request has a stable correlation ID. | UNIQUE constraint on correlation_id. |
| I6 | Every transition emits an audit event. | DB-level trigger on `provisioning_jobs` INSERT or UPDATE. Service layer is not involved in audit logging. |
| I7 | Retry increments the attempt counter. | The retry RPC atomically increments attempt_count. |
| I8 | Failure history is preserved. | ProvisioningFailure records are INSERT-only. No DELETE, no UPDATE. |
| I9 | Authorization is enforced exclusively through Platform Office rules. | All RPCs call `is_platform_operator()` — no ERP role checks. |
| I10 | Service wrappers contain no business logic. | Domain rules are in RPCs and DB constraints only. Service layer is a thin pass-through. |
| I11 | Stage ordering is sequential. | Stage N cannot start until stage N-1 has status SUCCEEDED. |
| I12 | Every provisioning request has exactly one job. | One-to-one: ProvisioningRequest → ProvisioningJob. No fan-out. |

### 5. Authorization Model

All provisioning operations are gated by `is_platform_operator()`:

| Operation | Required Role | Notes |
|-----------|---------------|-------|
| Create provisioning request | Any platform operator | Logging the request is non-destructive |
| Cancel a request/job | `owner` | Cancellation is a privileged action |
| Retry a failed job | `owner` or `operations` | Operations role can trigger retries |
| View provisioning status | Any platform operator | Read-only observability |
| View failure history | Any platform operator | Debugging and monitoring |
| Override provisioning config | `owner` | High-privilege override |

Authorization is checked:
1. At RPC entry — before any mutation
2. Not in the service layer — services are thin wrappers

### 6. Operational Metadata

Every provisioning request and job carries the following metadata:

| Field | Domain | Purpose |
|-------|--------|---------|
| correlation_id | ProvisioningRequest | Stable, caller-generated ID for idempotency and cross-system tracing |
| request_id | ProvisioningJob | Links job back to the originating request |
| requested_by | Both | Who initiated the provisioning |
| current_owner | ProvisioningJob | Which operator is currently handling this (nullable) |
| completed_by | ProvisioningJob | Who completed or cancelled the job (nullable) |
| requested_at | Both | When the request was created |
| started_at | ProvisioningJob | When execution first began |
| completed_at | ProvisioningJob | When execution ended (success, failure, or cancellation) |
| retry_count | ProvisioningJob | How many times this request has been retried |
| failure_reason | ProvisioningFailure | Human-readable description of the failure |
| failure_code | ProvisioningFailure | Machine-readable failure category |
| stage_name | ProvisioningStage | Which pipeline stage is or was active |
| stage_output | ProvisioningStage | Stage-specific result data (jsonb) |

This metadata is designed to support:
- **Auditing:** Complete provenance chain from request to completion
- **Debugging:** Full failure history with codes and reasons
- **Monitoring:** Counts by status, stage, and failure code
- **Incident response:** Correlation ID links provisioning events to other platform events
- **Dashboards:** Real-time and historical provisioning pipeline health

### 7. RPC Contract Specification

#### 7.1 `create_provisioning_request`

| Aspect | Specification |
|--------|---------------|
| Input | `p_correlation_id uuid`, `p_target_type text`, `p_target_config jsonb` |
| Output | `request_id uuid` |
| Authorization | Any platform operator (`is_platform_operator(auth.uid())`) |
| Validation | `p_target_type` must be non-empty; `p_target_config` must be valid JSON; correlation_id must be unique |
| Idempotency | If correlation_id exists → return existing request_id (no-op) |
| Failure semantics | Raises exception on duplicate correlation_id or invalid config |

#### 7.2 `retry_provisioning`

| Aspect | Specification |
|--------|---------------|
| Input | `p_request_id uuid`, `p_reason text` |
| Output | `job_id uuid` |
| Authorization | `is_platform_operator(auth.uid(), 'operations')` — owner or operations |
| Validation | Last job must be in FAILED state; must not be CANCELLED |
| Idempotency | Not idempotent — each call creates a new job with incremented attempt |
| Failure semantics | Raises exception if job is CANCELLED, SUCCEEDED, or still active |

#### 7.3 `cancel_provisioning`

| Aspect | Specification |
|--------|---------------|
| Input | `p_request_id uuid` |
| Output | `void` |
| Authorization | `is_platform_operator(auth.uid(), 'owner')` — owner only |
| Validation | Job must be in REQUESTED or QUEUED state |
| Idempotency | Idempotent — calling again on already CANCELLED returns success |
| Failure semantics | Raises exception if job is PROVISIONING, SUCCEEDED, or FAILED |

#### 7.4 Read operations

Read operations go through direct table queries in the service layer, not through RPCs. This matches the existing codebase pattern (see `getProvisioningStatusCounts`, `getAllWorkspaces`, `getActiveIncidents` — all query tables directly via the Supabase client).

RPCs are reserved for write operations that require:
- Authorization enforcement (`is_platform_operator()`)
- State validation (legal transition checks)
- Atomic multi-table mutations

Read-only queries to public-schema tables are permitted directly from the service layer per AGENTS.md §1.

| Service Function | Table | Authorization |
|-----------------|-------|---------------|
| `getAllProvisioningRequests()` | `provisioning_requests` | Any platform operator |
| `getProvisioningJobs(requestId?)` | `provisioning_jobs` | Any platform operator |
| `getProvisioningFailures(requestId?)` | `provisioning_failures` | Any platform operator |

### 8. Idempotency & Concurrency

#### 8.1 Duplicate Request Handling

- `create_provisioning_request` uses `correlation_id` as the idempotency key
- On duplicate correlation_id, return the existing `request_id` instead of creating a new request
- This allows safe retries of the create operation from the caller without risk of duplicate provisioning

#### 8.2 Concurrent Provisioning Protection

- A unique partial index on `provisioning_jobs (target_type, target_id) WHERE status IN ('QUEUED', 'PROVISIONING')` prevents two active jobs for the same target
- Concurrent dispatch attempts for the same target will hit a unique constraint violation, which the RPC catches and surfaces as a descriptive error

#### 8.3 Retry Behaviour

- Each retry creates a new `ProvisioningJob` row for the same `ProvisioningRequest`
- The `attempt_count` is incremented atomically from the previous job's count
- Failure history from all prior attempts is preserved in `ProvisioningFailure`
- Retry is only allowed from FAILED state — never from CANCELLED or SUCCEEDED

#### 8.4 Idempotent Operations

| Operation | Idempotent? | Key |
|-----------|-------------|-----|
| create_provisioning_request | Yes | correlation_id |
| cancel_provisioning | Yes (on same state) | request_id + state check |
| get_provisioning_status | Yes | N/A (read-only) |
| get_provisioning_failures | Yes | N/A (read-only) |
| get_provisioning_counts | Yes | N/A (read-only) |
| retry_provisioning | No | Each call creates a new job |

#### 8.5 Ownership Transfer

- `current_owner` on `ProvisioningJob` can be reassigned via the `assign` provisioning action
- Only platform operators with `owner` role can assign ownership
- Ownership transfer does not change the job's state machine status

### 9. Service Layer

Before introducing any new abstractions, audit the existing service layer in `src/lib/services/`.

#### 9.1 Existing Pattern Analysis

All existing service functions use `Promise<boolean>` for mutation results:

```typescript

```

Read functions use empty defaults on failure:

```typescript
// provisioning-service.ts — empty defaults on failure

export async function getFailedProvisioning(): Promise<FailedProvisioning[]>
```

#### 9.2 Convention Decision

**Preserve `Promise<boolean>` for mutations** — the project consistently uses this pattern. Do not introduce `ServiceResult<T>` or `{ success: boolean; error?: string }` for provisioning. If a future milestone adds error surfacing, extend uniformly across all services at that point, not in isolation for provisioning.

**Preserve empty-default arrays/objects for reads** — on query failure, return empty defaults.

#### 9.3 New Service Functions

All provisioning service functions follow existing conventions:

```
provisioning-service.ts (existing, extend)
├── getProvisioningStatusCounts()                               → already exists
├── getFailedProvisioning()                                     → already exists
├── createProvisioningRequest(targetType, config)               → Promise<boolean>
├── retryProvisioning(requestId, reason)                        → Promise<boolean>
├── cancelProvisioning(requestId)                               → Promise<boolean>
├── getAllProvisioningRequests()                                → ProvisioningRequest[]
├── getProvisioningJobs(requestId?)                             → ProvisioningJob[]
└── getProvisioningFailures(requestId?)                         → ProvisioningFailure[]
```

Service functions must:
- Call the RPC (not query tables directly when RPCs exist)
- Return `Promise<boolean>` for mutations
- Return empty defaults for reads
- Contain zero business logic (no state checks, no validation)

/**
 * Create a new provisioning request.
 * Returns true on success, false on failure.
 */
export async function createProvisioningRequest(
  correlationId: string,
  targetType: string,
  targetConfig: Record<string, unknown>,
): Promise<boolean> {
  const supabase = await getSupabase();
  if (!supabase) return false;

  try {
    const { error } = await supabase.rpc("create_provisioning_request", {
      p_correlation_id: correlationId,
      p_target_type: targetType,
      p_target_config: targetConfig,
    });
    return !error;
  } catch {
    return false;
  }
}

/**
 * Retry a failed provisioning request.
 * Returns true on success, false on failure.
 */
export async function retryProvisioning(
  requestId: string,
  reason: string,
): Promise<boolean> {
  const supabase = await getSupabase();
  if (!supabase) return false;

  try {
    const { error } = await supabase.rpc("retry_provisioning", {
      p_request_id: requestId,
      p_reason: reason,
    });
    return !error;
  } catch {
    return false;
  }
}

/**
 * Cancel a pending or queued provisioning request.
 * Returns true on success, false on failure.
 */
export async function cancelProvisioning(requestId: string): Promise<boolean> {
  const supabase = await getSupabase();
  if (!supabase) return false;

    const { error } = await supabase.rpc("cancel_provisioning", {
      p_request_id: requestId,
    });
    return !error;
  } catch {
    return false;
  }
}

/**
 * Fetch all provisioning requests for the operator console.
 * Returns empty array on failure.
 */
export async function getAllProvisioningRequests(): Promise<ProvisioningRequest[]> {
  const supabase = await getSupabase();
  if (!supabase) return [];

  try {
    const { data, error } = await supabase
      .from("provisioning_requests")
      .select("id, correlation_id, target_type, target_config, status, requested_by, requested_at, created_at")
      .order("created_at", { ascending: false });

    if (error) return [];
    return (data || []).map((r) => ({
      id: r.id,
      correlationId: r.correlation_id,
      targetType: r.target_type,
      targetConfig: r.target_config,
      status: r.status,
      requestedBy: r.requested_by,
      requestedAt: r.requested_at,
      createdAt: r.created_at,
    }));
  } catch {
    return [];
  }
}

/**
 * Fetch provisioning jobs, optionally filtered by request.
 * Returns empty array on failure.
 */
export async function getProvisioningJobs(requestId?: string): Promise<ProvisioningJob[]> {
  const supabase = await getSupabase();
  if (!supabase) return [];

  try {
    let query = supabase
      .from("provisioning_jobs")
      .select("id, request_id, target_type, target_id, status, current_stage, attempt_count, owner_id, requested_by, completed_by, requested_at, started_at, completed_at, created_at, updated_at")
      .order("created_at", { ascending: false });

    if (requestId) {
      query = query.eq("request_id", requestId);
    }

    const { data, error } = await query;
    if (error) return [];
    return (data || []).map((j) => ({
      id: j.id,
      requestId: j.request_id,
      targetType: j.target_type,
      targetId: j.target_id,
      status: j.status,
      currentStage: j.current_stage,
      attemptCount: j.attempt_count,
      ownerId: j.owner_id,
      requestedBy: j.requested_by,
      completedBy: j.completed_by,
      requestedAt: j.requested_at,
      startedAt: j.started_at,
      completedAt: j.completed_at,
      createdAt: j.created_at,
      updatedAt: j.updated_at,
    }));
  } catch {
    return [];
  }
}

### Prerequisites (database layer, not in scope of this plan)

The following database tables must exist before the TypeScript service layer can function. They are specified here as architectural contracts but implemented separately as Supabase migrations:

- `public.provisioning_requests` — maps to `ProvisioningRequest` domain entity
- `public.provisioning_jobs` — maps to `ProvisioningJob` domain entity
- `public.provisioning_stages` — maps to `ProvisioningStage` domain entity
- `public.provisioning_failures` — maps to `ProvisioningFailure` domain entity
- RPCs: `create_provisioning_request`, `retry_provisioning`, `cancel_provisioning`
- DB trigger: audit event emission on `provisioning_jobs` state transitions

Table definitions and RPC implementations are derived from the domain model and contracts in this document. `ProvisioningTarget` (§1.5) is reserved for Phase B multi-target support and does not require a dedicated table in Phase A.
